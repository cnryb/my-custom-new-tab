## Context

当前项目是基于 Vue 3 与浏览器扩展 API 的新标签页应用，现阶段已验证 `chrome.topSites` 和 `chrome.proxy.settings` 的基本能力，但页面结构仍是调试形态，缺少可持续扩展的模块边界。P0 需要在不引入额外后端服务的前提下，完成搜索、导航、代理控制和设置持久化四类基础能力，并保证新标签页首屏交互顺畅。

约束如下：

- 现有前端入口集中在 `src/App.vue`，需要为后续扩展预留组件化空间。
- 浏览器真实代理状态以 `chrome.proxy.settings` 为准，不能仅依赖本地缓存。
- 页面偏好、快捷网站等用户数据适合存储在 `chrome.storage.local`。
- P0 目标是形成稳定可用的首页，不引入 Microsoft To Do、工作区等后续能力。

## Goals / Non-Goals

**Goals:**

- 建立清晰的新标签页信息架构，形成搜索、快捷网站、最常访问、代理面板四个核心区域。
- 明确页面状态、浏览器状态与本地持久化数据的边界，降低后续扩展复杂度。
- 在现有浏览器 API 基础上完成可测试、可回归的 P0 功能定义。
- 为后续拆分组件、组合式逻辑和更多能力模块提供稳定基础。

**Non-Goals:**

- 不实现 Microsoft To Do、笔记、工作区、最近关闭标签页等 P1/P2 能力。
- 不在 P0 引入多代理配置集、PAC 脚本、自动切换规则等高级代理能力。
- 不实现跨设备同步、账号体系或远端配置服务。
- 不追求复杂视觉定制，仅保留满足日常使用的基础主题与布局偏好。

## Decisions

### 1. 采用“单页分区 + 后续可组件化”的页面结构

P0 首页按搜索框、快捷网站、最常访问、代理面板四个区块组织。初期可以仍由单个页面容器承载，但区块边界必须先被明确，以便后续拆分为独立组件与组合式逻辑。

选择原因：

- 能快速从调试页过渡到产品页。
- 避免 `App.vue` 继续同时承担 UI、数据获取和交互控制。

备选方案：

- 立即大规模组件化。此方案更整洁，但对当前项目阶段来说改动面偏大，不利于快速完成 P0。

### 2. 区分三类状态来源

P0 将状态分为三类：

- 页面偏好与快捷网站：保存在 `chrome.storage.local`
- 浏览器动态数据：通过 `chrome.topSites` 实时读取
- 代理真实状态：通过 `chrome.proxy.settings` 实时读取

选择原因：

- 可以避免把“用户草稿配置”和“浏览器当前实际状态”混为一谈。
- 能在刷新页面后恢复用户偏好，同时保持代理状态展示真实可信。

备选方案：

- 所有数据统一保存在本地设置中。该方案实现简单，但会导致代理状态与真实浏览器状态不一致。

### 3. 代理能力采用“草稿配置 + 实际应用状态”模型

代理模块分为两部分：

- 代理草稿配置：用户在页面上编辑的 `scheme`、`host`、`port`、`bypassList`
- 实际代理状态：由 service worker 调用 `chrome.proxy.settings` 读取和写入

选择原因：

- 用户可以保留常用表单值，而不强制要求其始终与当前浏览器状态一致。
- 便于实现“编辑后应用”和“一键关闭代理”两种行为。

备选方案：

- 页面表单始终直接绑定真实状态。该方案直观，但在清除代理后会丢失用户常用配置，不利于重复使用。

### 4. 搜索框采用统一输入解析流程

搜索框在提交时先判断输入是否为 URL 或可补全为 URL，否则按当前默认搜索引擎拼接搜索地址执行跳转。默认支持少量搜索引擎选项，并将选择结果持久化。

选择原因：

- 新标签页的首要任务是“快速开始”，统一输入框最符合使用习惯。
- 限制搜索引擎范围可以降低 P0 配置复杂度。

备选方案：

- 分成“地址栏模式”和“搜索模式”两个入口。该方案可控性高，但会增加用户决策成本。

### 5. 最常访问网站作为动态补充，不与快捷网站混合存储

最常访问网站保持只读动态数据源，展示前需做异常过滤、标题处理以及与快捷网站去重。用户自定义快捷网站则保持独立管理。

选择原因：

- 动态推荐与用户固定入口职责不同，混合后容易造成重复和管理混乱。
- 去重后首页信息密度更合理。

备选方案：

- 将最常访问网站直接合并进快捷网站列表。该方案实现简单，但会破坏固定入口的稳定性。

## Risks / Trade-offs

- [首页状态来源增多] → 通过状态分层和清晰命名区分“设置”“动态数据”“真实代理状态”。
- [代理 API 在不同环境下行为不一致] → 在 P0 中优先定义读取、设置、清除三条稳定路径，并为失败场景保留反馈。
- [页面仍集中在少量文件中，后续维护困难] → 在设计上先固定模块边界，实施时优先提取复用逻辑与类型定义。
- [最常访问站点质量不稳定] → 增加过滤、数量限制与隐藏开关，避免影响主界面可读性。

## Migration Plan

该变更属于本地扩展页面升级，不涉及远端部署和数据迁移。实施时建议按“页面骨架 -> 设置存储 -> 快捷网站 -> 搜索框 -> 最常访问 -> 代理面板”顺序推进；若中途需要回退，可保留当前最小页面并逐步切换功能入口。

## Open Questions

- 搜索结果默认在当前标签页打开还是新标签页打开，是否需要提供设置开关。
- 快捷网站排序在 P0 中采用“上移/下移”还是直接提供拖拽。
- 主题能力在 P0 中是否只保留深浅两种模式，还是允许背景自定义。
